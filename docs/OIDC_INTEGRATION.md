# OIDC / OAuth2 Integration Guide

The AmneziaWG Captive Portal acts as an **OIDC Relying Party (Client)**. It delegates the authentication of VPN users to an external Identity Provider (IdP) such as Authelia, Keycloak, Auth0, Google Workspace, or Authentik.

This guide explains how to configure any generic OIDC provider and connect it to this portal.

## 1. Configuring the Identity Provider (IdP)

Regardless of which IdP you use, you must create a new "Application" or "OIDC Client" in your provider's admin panel. 

You must configure the following settings on your IdP:

*   **Client ID**: A unique identifier for the portal (e.g., `amnezia-auth`).
*   **Client Secret**: A secure, generated string used to authenticate the portal to the IdP.
*   **Client Authentication Method**: `client_secret_basic` or `client_secret_post`.
*   **Allowed Grant Types**: `Authorization Code` (`authorization_code`).
*   **Allowed Scopes**: The portal requests exactly three scopes: `openid`, `profile`, and `email`. Ensure your IdP grants these scopes.
*   **Redirect URIs (Callback URIs)**: This is the most critical setting. You must tell the IdP where to send the user *after* a successful login. 
    *   If you access the portal directly via IP, it will look like: `http://10.0.42.1:8000/auth` (Assuming `10.0.42.1` is your `vpn_server_ip`).
    *   If you expose the portal via a domain name (e.g., behind Traefik inside the VPN), it will look like: `https://vpn-auth.dreidel-local/auth`.
    *   **Note:** The path must always end exactly in `/auth`.

### Example: Authelia Configuration

If you are using Authelia, your `configuration.yml` under the `identity_providers.oidc.clients` section would look like this:

```yaml
identity_providers:
  oidc:
    clients:
      - id: 'amnezia-auth'
        description: 'AmneziaWG VPN Captive Portal'
        secret: '$pbkdf2-sha512$...' # Hashed version of your raw secret
        public: false
        authorization_policy: 'two_factor' # Enforce 2FA for VPN access
        redirect_uris:
          - 'http://10.0.42.1:8000/auth'
        scopes:
          - 'openid'
          - 'profile'
          - 'email'
        grant_types:
          - 'authorization_code'
```

## 2. Configuring the Captive Portal (Ansible)

Once the application is created in your IdP, you need to copy three values into your Ansible configuration.

### In `group_vars/vpn_servers/vars.yml`

*   **`oidc_discovery_url`**: This is the URL where the portal can auto-discover your IdP's endpoints. It almost always ends in `/.well-known/openid-configuration`.
    *   *Authelia example*: `https://auth.yourdomain.com/.well-known/openid-configuration`
    *   *Keycloak example*: `https://keycloak.yourdomain.com/realms/master/.well-known/openid-configuration`

### In `group_vars/vpn_servers/vault.yml`

*   **`vault.oidc_client_secret`**: The raw, unhashed secret generated by your IdP.
*   *(Optional)* If you didn't hardcode `oidc_client_id` in `vars.yml`, ensure your client ID matches what is in the IdP. By default, the template assumes the variable `oidc_client_id` is passed (e.g., `amnezia-auth`).

## 3. How the Flow Works (For Troubleshooting)

1. Unauthenticated VPN user navigates to `http://10.0.42.1:8000`.
2. The portal redirects them to the IdP's `/authorize` endpoint.
3. User performs 2FA/SSO login on the IdP.
4. IdP redirects the user back to `http://10.0.42.1:8000/auth?code=XYZ`.
5. The portal exchanges the `code=XYZ` for an Access/ID Token at the IdP's `/token` endpoint using the `oidc_client_secret`.
6. The portal reads the user's information from the token, logs the successful login, and executes `iptables` to grant network access.
